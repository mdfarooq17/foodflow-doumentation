1.3. Project Introduction

FoodFlow is a modern, responsive, full-stack food delivery and earnings management system built on stateless REST interfaces. The application solves critical real-time operational coordination friction between multiple roles: Customers, Restaurant Owners, Riders, and Central Administrators.

Unlike conventional delivery setups plagued by high network latency and un-reactive components, FoodFlow implements an asynchronous state polling architecture that pushes updates dynamically across portals, eliminating manual browser refreshes.

Key Innovations: Stateless JWT role-based claims verification, dynamic driver dispatch pipelines, live dashboard kitchen notifications with audio-visual triggers, star review aggregation models, and premium glassmorphic visual presentation.

1.4. Core Objectives

The system was designed and verified against four primary development objectives:

  1. Architectural Cohesion: Bridging user interactions via role-based claims decoded immediately from unified cryptographic authorization tokens.
  2. Algorithmic Integrity: Calculating real-time average store feedback (1-decimal place precision) and allocating automated Rs. 150 transaction fees to delivery drivers.
  3. Performance Optimization: Tuning build stylesheet budgets to permit custom dark dashboards rendering without bundle compilation failures.
  4. Security Rigor: Blocking cross-portal injection attacks, database credential theft, and invalid template memory reference bugs.

2. System Design (SDD) Specifications

Database, Directory, and Core API Architectures

2.1. Executive Summary & Problem Statement

FoodFlow solves modern food delivery friction by providing a highly synchronized, multi-portal application built on stateless rest interfaces. The architecture bridges the real-time operational gap between Customers, Restaurants, Riders, and Admins.

Traditional food platforms experience high latency and severe coordination overhead caused by siloed operational frameworks. Customers suffer from static, un-reactive delivery trackers, restaurants struggle with complex manual rider dispatch coordination, and riders lack micro-incentives like dynamic wallet ledgers that track real-time delivery counts and earnings.

FoodFlow addresses these limits by establishing a clean, unified platform:

  • Unified Cryptographic Identities: Single-sign-on layout where user authentication claims are decoded to securely direct users to their dedicated portals.
  • Real-time Polling Engine: Dashboards and active tracking templates trigger asynchronous checks (every 5 seconds) to cycle order status, update dispatch queues, and trigger alerts without page reloads.
  • Low Overhead State Lifecycle: Strict state transition sequences powered by stateless JSON Web Token (JWT) assertions.

2.2. Functional Scope & Portals

2.2.1. Customer Portal Workspace

Customers access a premium, dark-themed responsive discovery panel. Features include restaurant filtering by active municipal boundaries, interactive cart checkouts featuring auto-calculated grand totals (adding a fixed Rs. 150 delivery fee), a persistent graphical order-status bar, and a star feedback system.

2.2.2. Restaurant Kitchen Dashboard

Kitchen operators can configure restaurant details, manage menu items within food categories, and process incoming orders. When an order arrives, a prominent visual alert appears. Ready orders can be dispatched by selecting from a list of active online riders.

2.2.3. Rider Fleet Ledger

Riders operate a dashboard featuring a toggle to control their online availability. When set to online, riders become eligible for manual restaurant dispatch. Delivering an order increments the rider's successful order count and credits Rs. 150 to their earnings wallet.

2.2.4. Central System Governance

Root Administrators monitor real-time system metrics (users, active stores, order count, and gross system revenue). Admins verify and approve new restaurants, define system cities, and manage operational categories.

2.6. Order Operational Flow

The system relies on a secure transactional loop from order placement to customer feedback:

  1. Order Initiation: Customer completes checkout. The order is recorded as pending and added to the restaurant's active list.
  2. Order Acceptance: Restaurant receives an alert, accepts the order (changing status to accepted), and advances it to preparing and then ready.
  3. Rider Allocation: Restaurant lists active online riders, chooses an available rider, and executes a manual dispatch.
  4. Delivery Phase: Assigned rider picks up the package (out-for-delivery) and marks it as delivered upon arrival.
  5. Wallet & Review Updates: Delivering the order automatically adds Rs. 150 to the rider's earnings wallet, increments their delivery count, and unlocks the customer feedback star modal to dynamically recalculate the restaurant's average rating in the database.

2.7. Future Enhancements

  • Bidirectional WebSockets: Replace dynamic polling with a persistent event network built on Socket.io to reduce server overhead.
  • Secure Payment Gateway: Add credit/debit card support using Stripe Elements with server-side webhook validation.
  • Interactive Map Tracking: Integrate a map interface showing distances between restaurant locations and customer delivery addresses.

6. Security Architecture & Resolved Bug Fixes

Active Protections, Vulnerabilities mitigation, and compiler bugs fixes

6.1. Implemented Architectural Controls

Bcryptjs Hashing (10 Rounds)

Passwords are salted and hashed before database commits, preventing raw credentials visibility from dictionary crack matching database tables.

Token RBAC Interceptor middle-tier

Private endpoints check user role claim variables in decrypted JWT tokens to prevent unauthorized cross-portal API requests.

Query Projection Constraints

Database operations exclude password hashes (e.g. User.find({}, '-password')) to prevent accidental credential leakage in JSON responses.

6.2. Threat Modeling & Vulnerability Mitigation Plan

We analyzed the current system structure against common vulnerabilities and defined clear mitigation plans:

1. Client-Side JWT Storage (XSS Risk)

Vulnerability: Access tokens are currently saved in browser localStorage, making them susceptible to theft via Cross-Site Scripting (XSS) attacks.

Mitigation Plan: Migrate token storage from localStorage to **HttpOnly, Secure Cookies**, which prevents access via JavaScript and neutralizes script-based session theft.

2. Lack of Authentication Rate Limiting

Vulnerability: The login route permits unlimited connection attempts, exposing user accounts to automated dictionary and brute-force cracking tools.

Mitigation Plan: Integrate an active rate limiter like express-rate-limit to restrict logins to a maximum of 5 attempts within a 15-minute window per IP.

3. Absence of Strict Input Schema Validation

Vulnerability: Loose schema definitions allow potential attackers to pass NoSQL operator objects (such as $ne) to query fields.

Mitigation Plan: Enforce strict request body structures using validators like Joi or express-validator before executing queries.

6.3. Strict TypeScript Compiler Bug Fixes

Major bugs resolved to allow strict template checks compilation during rating modals additions:

Safe Navigation Operators (?.): TS2532 Resolved

Corrected raw nested sub-document queries (order.review.rating) to optional chaining navigators (order.review?.rating), preventing compilation crashes during loading states when reviews are undefined.

Interface Optional Properties Declarations: TS2339 Resolved

Updated client-side inline Order interface models in typescript components to explicitly declare optional review schemas properties.

Standard Attribute Style Binding (NG5002 Resolved)

Resolved Angular parser error by updating typo event property binding syntax (style)="cursor:pointer;" to standard HTML style layouts declaration: style="cursor:pointer;".
